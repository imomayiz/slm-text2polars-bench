# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

A polars-bench hackathon submission: FastAPI + Hugging Face Transformers service that generates Polars DataFrame code from natural-language questions. Optimized for the scoring formula `Score = N / (T * VRAM^0.1 * RAM^0.01)` where N=correct answers, T=wall time.

## Commands

```bash
# Install (no requirements.txt currently checked in — needs transformers, accelerate, polars, torch, fastapi, uvicorn, requests)
pip install transformers accelerate polars torch fastapi uvicorn requests

# Run the API server
uvicorn main:app --host 0.0.0.0 --port 8000

# Client smoke test (requires running server)
python test_api.py --url http://localhost:8000

# Generate the gold benchmark file
python make_starter_bench.py          # writes bench.json

# End-to-end smoke test (executor + comparator, no GPU needed — uses reference code)
python smoke_test.py

# Run benchmark against a model (requires GPU)
python run_bench.py --bench bench.json --model Qwen/Qwen2.5-Coder-7B-Instruct
python run_bench.py --bench bench.json --model <model> --repair   # with error-feedback repair loop

# Sweep multiple models
python sweep.py --bench bench.json --profile 24gb

# Docker
docker build -t polars-bench-sub .
docker run --gpus all -p 8000:8000 polars-bench-sub
```

## Architecture

Two parallel code paths exist for the same task (code generation from NL questions):

### API Server Path (`main.py` → `inference.py`)
- `main.py`: FastAPI app with `/health`, `/generate`, `/generate_batch` endpoints. Model loads once at startup via lifespan context manager. Config via env vars (MODEL, DTYPE, MAX_MODEL_LEN, GPU_MEM_UTIL, MAX_TOKENS).
- `inference.py`: `PolarsInferenceEngine` loads `AutoModelForCausalLM` + `AutoTokenizer` (Hugging Face). Handles prompt building, chat-format few-shot construction, and output cleaning. Supports both structured requests (question+schema) and raw prompt passthrough.

### Offline Bench Path (`run_bench.py` → `generator.py` → `executor.py`)
- `generator.py`: `PolarsGenerator` — MLX on Apple Silicon for batch evaluation. Has `generate()` for one-shot and `generate_with_repair()` for generate→execute→retry-on-error loop. Duplicates prompt/few-shot logic from `inference.py` (intentionally separate).
- `executor.py`: Sandboxed subprocess executor for model-generated code. Serializes DataFrames via parquet over stdin, execs code in isolation, returns results. Also contains `outputs_match()` for tolerant comparison (handles row order, dtype casting, float tolerance).
- `run_bench.py`: End-to-end benchmark runner. Loads `bench.json`, runs generator, executes outputs, compares to gold, computes official score.
- `sweep.py`: Multi-model sweep runner. Each model runs in a fresh subprocess. Produces `sweep_results/leaderboard.csv`.

### Bench Format
`bench.json` (generated by `make_starter_bench.py`) contains items with: question, schema, inline fixture data, reference_code, expected_output, and tolerance settings.

## Key Constraints

- `inference.py` and `generator.py` both have their own `SYSTEM_PROMPT`, `FEW_SHOT`, `build_messages`/`build_prompt`, and `clean_code` — they are intentionally decoupled. Changes to prompt engineering must be mirrored if both paths should stay in sync.
- Run uvicorn single-process (`--workers 1`). Multi-worker uvicorn will OOM by loading multiple model copies.
- Generated code must assign its answer to a variable named `result`.
- `executor.py` runs generated code in a subprocess with `pl` and named DataFrames in scope — no other imports available.
